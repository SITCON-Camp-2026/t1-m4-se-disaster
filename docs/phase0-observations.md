# Phase 0 Observations

## 我們做出了什麼？

- 建立了一個可展示的災害資訊整理工作台，首頁能看到原始資訊與整理草稿。
- 提供了可編輯草稿欄位，讓使用者留下候選類型、摘要、人工確認備註與「不能直接變成任務」標記。
- 把資料品質問題從原始資訊中明確標成需要人工確認，而不是假裝已確認。

## 哪些資料讓 agent 很容易整理？

- M-010 這類現場更新看起來很完整，適合被整理成地點狀態候選。
- M-003 與 M-009 也比較容易整理成現場資訊與資源狀態的候選內容。

## 哪些資料讓 agent 亂補或誤判？

- M-001、M-004、M-007 這些社群貼文很容易被誤當成已確認事實。
- M-012 來電訊息沒有足夠的當事人或現場驗證，不能被直接當成任務。
- M-006 兩種相互衝突的說法顯示現場資訊可能不一致，需要人工確認。

## 哪些資訊不能直接變成任務？

- M-001：缺少精確地點與確認來源，不能直接派人。
- M-004：訊息來自社群，且沒有實際可執行的現場確認。
- M-011：涉及長者隱私與位置不明，不能直接公開或建立任務。
- M-012：來電者不是當事人，缺乏確認機制。

## 哪些資訊的操作者不是當事人？

- M-012 的來電者是家屬，不是當事人。
- M-011 的訊息來自志工代轉述，並非直接由當事人發出。
- M-007 的社群貼文和留言也不代表真正的執行者或現場責任人。

## 我們剛剛偷偷假設了什麼？

- 我們一開始可能會假設「看起來像任務」就能直接派工，這是錯的。
- 我們也可能假設來源是官方公告就代表已驗證，這也不安全。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要更完整的來源驗證流程。
- 需要把每筆資訊的確認者、確認時間與可執行性一起呈現。
- 需要讓草稿能與正式任務流程連接，而不是停留在候選階段。

## 我們需要更清楚資料格式的地方

- 每筆資訊應該有「確認者」「確認時間」「地點」「是否可公開」等欄位。
- 原始資料與整理草稿應該分開，避免把未確認內容誤當成正式資料。

## 我們需要人工確認的地方

- 社群貼文是否真實、是否仍有效。
- 現場回報之間是否衝突。
- 是否存在個資或需要保護的資訊。
- 是否有足夠的現場資訊才能建立任務。
