# 訪談紀錄（Release 01）

這份文件整理了三種不同觀點的使用者訪談，分別代表回報者、資訊整理者與行動者。每一段都以使用者角度說明他們想完成什麼、看不懂什麼、最怕什麼，以及最在意的資訊安全問題。

## 1. 回報者觀點

1. 我想完成的事
   - 想把我看到或聽到的訊息快速送出去。
   - 想在不確定時先留下線索，而不是被表單卡住。

2. 我看不懂的地方
   - 不知道哪些欄位要填「我確定」，哪些要填「我不確定」。
   - 不知道系統會不會把我的轉述，誤當成我親眼確認。

3. 我最怕的錯誤
   - 我的轉述被當成事實。
   - 系統把模糊內容自動補成任務或已確認狀態。

4. 我不敢直接相信的資訊
   - 社群貼文、家屬代述、口耳相傳、沒有明確來源的訊息。
   - 缺少地點、時間與確認者的資訊。

5. 我希望系統提醒我的事
   - 這是我看到的，還是我親眼確認的。
   - 哪些欄位還需要人工確認。

6. 我想反問開發者的一個問題
   - 如果我只知道一半，系統能不能讓我保留不確定，而不是逼我猜？

## 2. 資訊整理者觀點

1. 我想完成的事
   - 想把原始資訊整理成下一位協作者能理解的結構，不想讓不確定內容看起來像已確認事實。
   - 想把哪些資訊應該先標成「需要人工確認」，哪些內容只能先保留為候選，講得更清楚。

2. 我看不懂的地方
   - 目前畫面沒有把「資訊取得方式」與「查核狀態」分開，容易讓人把來源當成可信度。
   - 不清楚哪些內容是原始資訊、哪些是整理草稿、哪些是 AI 推測。

3. 我最怕的錯誤
   - 把社群貼文、口述轉述或沒有驗證的回報，直接整理成任務。
   - 讓 AI 自動補出原文沒有的地點、時間、需求類型或確認者，造成假象。

4. 我不敢直接相信的資訊
   - 互相衝突的現場回報。
   - 來自非當事人、沒有確認者、沒有時間戳或來源不明的資訊。
   - 看起來像任務，但其實不足以支撐實際行動。

5. 我希望系統提醒我的事
   - 每筆資訊的原始文字要保留下來，方便追溯與核對。
   - 哪些內容已經過人工確認，哪些只是候選，哪些不能直接變成任務。
   - 為什麼我會做出這個判斷，避免下一位協作者只看結論不看原因。

6. 我想反問開發者的一個問題
   - 如果一筆資訊看起來像任務，但目前沒有足夠確認，系統要怎麼明確防止它被誤當成可執行任務？

## 3. 行動者觀點

1. 我想完成的事
   - 想知道哪些資訊現在可以採取行動，哪些還不能動。
   - 想在出發前知道這筆資訊是否還有效、是否已經過時或衝突。

2. 我看不懂的地方
   - 不清楚目前顯示的內容是已確認、待確認，還是只是候選。
   - 看不出某筆資訊是不是已經被確認過，還是只是被整理過一次。

3. 我最怕的錯誤
   - 把未確認資訊當成已確認，跑錯地方或做錯事。
   - 看到看似完整的內容，卻忽略它其實已經過時或與其他消息有衝突。

4. 我不敢直接相信的資訊
   - 缺少精確地點、確認者或最新狀態的資訊。
   - 來自轉述、社群、或沒有明確執行者的資訊。

5. 我希望系統提醒我的事
   - 這筆資訊是否還有效。
   - 是否需要再確認，是否應該先停在候選階段。

6. 我想反問開發者的一個問題
   - 如果我現在不能確定這筆資訊是否能用，畫面要怎麼讓我不會誤判？

## 4. 這些觀點的共同發現

- 三個角色都很在意「這筆資訊現在能不能安全採用」，而不只是「資訊看起來是不是整理得漂亮」。
- 目前原型已經把「不能直接變成任務」的概念說得比較清楚，但還需要更清楚地把「來源方式」、「查核狀態」與「行動可行性」分開。
- 這些回饋沒有直接要求完整功能清單，而是反覆提醒：不要讓不確定資訊看起來像已確認事實。
